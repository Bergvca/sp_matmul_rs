## What this crate is

`sp_matmul_rs` is a Rust port of [`sparse_dot_topn`](https://github.com/ing-bank/sparse_dot_topn): sparse CSR × CSR matrix multiplication with **fused top-n selection per row**. It ships as both a Rust crate (crates.io) and a Python extension built with PyO3 + maturin (PyPI).

The dominant performance lever is **L1/L2 cache blocking**: the default driver column-chunks `B` so the dense per-row accumulator and streamed B fragments stay resident in L1d. This is built in (in `chunked.rs`), not optional.

## Common commands

### Rust

```shell
cargo build --release                     # default features: rayon
cargo build --no-default-features         # sequential-only build
cargo test                                # runs unit + integration tests
cargo test --no-default-features          # tests without rayon (matches CI)
cargo test --features rayon               # explicit rayon (matches CI)
cargo test --test parity                  # one integration test binary
cargo test threading_invariance::case_X   # one test by name
cargo bench                               # criterion benches in benches/tfidf_like.rs
cargo fmt --all -- --check                # CI gate
cargo clippy --all-targets --all-features -- -D warnings  # CI gate
```

### Python extension

```shell
pip install maturin
maturin develop --release --features python,rayon   # build & install into current venv
pytest tests/python -q                              # Python test suite
python -c "import sp_matmul_rs; print(sp_matmul_rs.kernel_info())"  # check detected cache + defaults
```

The Python tests require the extension to be (re)built via `maturin develop` after any Rust change.

### Goldens

Parity tests under `tests/parity.rs` consume `.npz` goldens checked into `tests/data/`. They are generated by the legacy C++ `sparse_dot_topn` extension via `tests/data/generate_goldens.py` — do not regenerate from this crate's output (it would defeat the purpose of the test).

## Architecture

### Crate layout (`src/`)

Module responsibilities, from leaves up:

- `scalar` / `index` — traits + impls for the supported value (`f32, f64, i32, i64`) and index (`i32, i64`) dtypes. Every public kernel is generic over `<V: Scalar, I: Index>`.
- `csr` — `CsrView<'a, V, I>` (borrowed, wraps numpy buffers) and `CsrMatrix<V, I>` (owned, the return type).
- `maxheap` — bounded max-heap for retaining top-n per row.
- `matmul` — `sp_matmul` (no top-n). Two-pass: size then fill, with a dense accumulator + linked-list of touched columns.
- `matmul_topn` — public `sp_matmul_topn` entry point. Defines `TopNOptions<V>` and `SortMode`. Delegates to the chunked driver; the unchunked path is retained only as a test oracle (`sp_matmul_topn_unchunked_for_tests`, `#[doc(hidden)]`).
- `chunked` — **the cache-blocking pillar.** Column-chunks B, runs one A-row across all chunks while keeping a single per-row `MaxHeap` of size `top_n`. Per-row scratch (`sums`, `next[]`) is sized to `chunk_cols`, not `ncols`. Includes the `default_chunk_cols` heuristic (L1d-derived; macOS `sysctl` path because the `cache-size` crate's CPUID approach returns `None` on Apple Silicon). Two B-row projection strategies (`BProjection::BinarySearch` vs `Cursor`) selected by `pick_projection`. Also hosts the row-blocked / tiled kernels (`row_block`, `tile_b`).
- `tiled` — `TiledB`: B re-bucketed per column-chunk with chunk-local `u16` indices (the O3 "Strategie C" optimization). Halves index bandwidth in the inner loop.
- `parallel` (feature `rayon`) — Rayon-backed driver. Unit of work is a slice of rows; the column-chunk loop stays serial within a worker so per-row scratch stays L1-hot. **Threading-invariant**: bit-identical to the sequential path for fixed inputs and fixed `(chunk_cols, projection)`. Builds a fresh `ThreadPool` per call so callers can invoke from inside their own pool without oversubscription. `n_threads = None | Some(1)` short-circuits to the sequential chunked driver — no rayon overhead.
- `zip` — `zip_sp_matmul_topn`, retained for the distributed/cluster use case where chunks are computed on separate machines. Single-node code should use the chunked driver instead.
- `python` (feature `python`) — PyO3 bindings exposed as the `sp_matmul_rs._core` cdylib. `python/conv.rs` borrows numpy CSR buffers as `CsrView`; `python/dispatch.rs` does the dtype dispatch. The user-facing Python API lives in `python/sp_matmul_rs/api.py` and wraps `_core`.

### Public API surface

Re-exported from `lib.rs`:
- Sequential: `sp_matmul`, `sp_matmul_topn` (drives chunked internally), `zip_sp_matmul_topn`.
- Types: `CsrView`, `CsrMatrix`, `Index`, `Scalar`, `SortMode`, `TopNOptions`, `AccumMode`, `BProjection`, `TiledB`.
- Knobs on `TopNOptions`: `chunk_cols`, `n_threads`, `accum_mode` (`Adaptive` default), `projection` (heuristic default), `row_block` (auto-tile default), `tile_b`.

### Python distribution

- `pyproject.toml` (`[tool.maturin]`) builds with `features = ["python", "rayon"]`, target module name `sp_matmul_rs._core`, Python source root `python/`.
- Imports as `sp_matmul_rs`. Does **not** share a namespace with `sparse_dot_topn` — both can be installed side-by-side. Surface (`sp_matmul`, `sp_matmul_topn`, `zip_sp_matmul_topn`) mirrors `sparse_dot_topn` so migration is a one-line import swap.
- The GIL is released around the kernel.

## Testing strategy

Three integration test binaries (each in `tests/`):

1. `parity.rs` — golden-output equivalence against the legacy C++ extension's `.npz` fixtures, for every supported `(V, I)` dtype pair. Tolerance is per-dtype.
2. `chunk_invariance.rs` — for every golden, run the chunked driver at multiple `chunk_cols` values (incl. 1, ncols, 10·ncols) and assert each matches the unchunked oracle. Pins the chunk-size correctness invariant.
3. `threading_invariance.rs` (feature-gated on `rayon`) — for every golden, assert `n_threads ∈ {2, 4, 8}` results match the sequential path bit-for-bit (per-row value-multiset for ints, per-dtype tolerance for floats).

`properties.rs` adds proptest-driven invariant checks on `f64 × i32` (the dtype-generic kernels are covered for other dtypes via `parity.rs`).

Python tests live in `tests/python/` and run via pytest after `maturin develop`.

## Conventions worth knowing

- Source files open with a `//!` module-doc explaining the design intent.
- The chunked kernel uses `u32` sentinels (`UNVISITED = u32::MAX`, `HEAD_NIL = u32::MAX - 1`) deliberately to keep `next[]` half the size of a `usize` version — the drain is a serial pointer chase and L1 residency dominates.
- Cargo manifest excludes test data and Python build artefacts from the published crate; check `[package].exclude` before adding large files at the repo root.
